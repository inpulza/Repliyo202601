Mi Auditoría del Plan de Replit
✅ Lo que está Perfecto:

La Estructura de Datos: Guardar conversation_summary y summary_last_message_id en la tabla Conversations es correcto y eficiente.

El Trigger Asíncrono: Generar el resumen después de responder ("background job") para no hacer esperar al usuario es una decisión técnica brillante.

El Enfoque Híbrido: [Resumen Histórico] + [Últimos 10 Mensajes Verbatim] es el estándar de oro.

⚠️ El Riesgo (El "Teléfono Descompuesto"): El agente propone:

"Primera vez: Resumir últimos 20 mensajes" "Próxima vez: Tomar resumen anterior + nuevos mensajes -> Nuevo resumen consolidado"

El Peligro: Si resumes un resumen, y luego resumes ese resumen, y así 10 veces... la información se degrada. Es como fotocopiar una fotocopia. Al final, "El cliente compró zapatos rojos talla 42 el martes" se convierte en "El cliente compró algo".

La Solución: Instrucciones de "Alta Fidelidad". Debemos instruir al LLM que genera el resumen para que preserve datos duros (entidades).

Instrucción: "Al resumir, mantén intactos nombres, fechas, números de teléfono, correos y productos específicos. Solo resume la 'paja' conversacional."

"Este plan de Memoria Progresiva (Fase 1) es excelente. El enfoque híbrido es exactamente lo que necesitamos para mantener el contexto sin disparar los costos.

APROBADO con una Mejora Crítica en el Prompt de Resumen:

El riesgo de resumir resúmenes es la pérdida de datos específicos (Efecto 'Teléfono Descompuesto'). Para mitigar esto, cuando implementes la función generateConversationSummary, asegúrate de que el System Prompt del Resumidor tenga estas instrucciones explícitas:

Preservación de Entidades: 'Tu prioridad absoluta es conservar DATOS DUROS: Nombres, Fechas, Números de teléfono, Correos electrónicos, Precios y Productos específicos mencionados.'

Estilo Telegráfico: 'No redactes prosa fluida. Usa un estilo denso y lleno de datos. Ejemplo: Cliente pregunta precio zapatos rojos -> Agente confirma $50 -> Cliente pide enlace de pago.'

No Opinión: 'No interpretes sentimientos, solo hechos.'

Plan de Ejecución:

DB: Modifica la tabla conversations con los 3 campos propuestos.

Backend: Crea el servicio de resumen con este prompt de "Alta Fidelidad".

💡 Por qué esta mejora es clave
Si tu IA olvida el número de teléfono que el cliente dio hace 3 días, el cliente se enfadará. Con esta instrucción de "Preservación de Entidades", conviertes el resumen en una mini base de datos de hechos, no solo en un cuento. ¡Es la diferencia entre un chat y un asistente real!
