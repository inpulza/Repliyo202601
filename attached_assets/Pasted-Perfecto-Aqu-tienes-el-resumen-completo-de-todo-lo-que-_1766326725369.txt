3️⃣ MESSAGE BUFFERING (ANTI-INTERRUPCIÓN)
El problema:
El sistema responde inmediatamente a cada mensaje. Si el usuario envía 3 mensajes seguidos ("Hola", "tengo una duda", "sobre taxes"), el agente interrumpe 3 veces.

La solución: Debounce
Usuario envía mensaje → Sistema espera 15-20 segundos
Si llega otro mensaje → Reinicia el timer
Cuando termina el timer → Envía TODOS los mensajes juntos al LLM
Implementación recomendada:
En memoria con Map + setTimeout (no Redis)
Solo aplica a DMs, comentarios siguen con flujo normal
Timer configurable por agente (ej: dmBatchDelaySeconds: 15)

5️⃣ MEMORIA PERSISTENTE: EXTRACCIÓN DE ENTIDADES
El problema:
Los resúmenes actuales se "diluyen" con el tiempo. Resumen de resumen de resumen = pérdida de datos importantes.

La solución: Nueva tabla conversation_user_entities
CREATE TABLE conversation_user_entities (
  id VARCHAR PRIMARY KEY,
  conversation_id VARCHAR NOT NULL,
  author TEXT NOT NULL,
  phone TEXT,
  email TEXT,
  real_name TEXT,
  income TEXT,
  dependents INTEGER,
  interests TEXT[], -- array
  key_dates JSONB,
  extracted_at TIMESTAMP,
  last_message_id VARCHAR
);
Flujo:
Llegan mensajes nuevos
LLM extrae entidades (teléfono, nombre, etc.) → Se guardan en tabla (NUNCA se pierden)
LLM genera resumen contextual → Puede cambiar, no importa
Al responder: El agente recibe ENTIDADES + RESUMEN
Beneficios:
Datos importantes nunca se pierden
Son buscables (puedes filtrar por teléfono, ingresos, etc.)
Son editables directamente
