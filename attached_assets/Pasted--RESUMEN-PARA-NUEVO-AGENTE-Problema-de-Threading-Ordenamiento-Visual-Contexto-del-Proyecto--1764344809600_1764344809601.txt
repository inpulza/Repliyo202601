⚠️ EL PROBLEMA ACTUAL DE UX
Descripción:
Las respuestas enviadas desde Repliyo no aparecen visualmente debajo del comentario al que respondieron.

Ejemplo concreto (TikTok):

Comentario de dra.melvavidal: "¿Cómo? ¿Respuesta de prueba?"
Nuestra respuesta: "@dra.melvavidal Perdone doctora!!"
Esperado: La respuesta debe aparecer INMEDIATAMENTE debajo del comentario de la doctora
Actual: Puede aparecer en otro lugar de la conversación
Causa raíz:

Nosotros guardamos: parentMessageId = c588b3e3 (el comentario de la doctora)
Metricool devuelve el mismo mensaje con: parentMessageId = d802cf25 (el post original)
Metricool NO conoce threading anidado - para ellos todo es "comentario del post"

🔍 Lógica Actual de Threading (Inbox.tsx líneas 227-262)
const threadMessages = useMemo(() => {
  // 1. Separar mensajes raíz (sin parentMessageId) de replies
  const rootMessages = messages.filter(m => !m.parentMessageId);
  const replies = messages.filter(m => m.parentMessageId);
  
  // 2. Ordenar raíces por timestamp
  rootMessages.sort((a, b) => timestamp_a - timestamp_b);
  
  // 3. Para cada raíz, insertar sus replies debajo
  for (const root of rootMessages) {
    orderedMessages.push(root);
    const rootReplies = replies.filter(r => r.parentMessageId === root.id);
    orderedMessages.push(...rootReplies);
  }
  
  // 4. Replies huérfanos (padre no encontrado) van al final
});

🎯 Lo que necesita verificarse/arreglarse

El problema de Metricool:

Metricool asigna parentMessageId diferente (del post, no del comentario)
Nuestro mensaje outbound tiene el parentMessageId correcto
La reconciliación preserva nuestro parentMessageId (no lo sobrescribe)
Pero puede haber edge cases donde el threading se rompe
Posibles soluciones a explorar:

Ignorar el parentMessageId que viene de Metricool para mensajes reconciliados
Usar el contenido del mensaje (menciones @username) para inferir el padre
Crear una heurística basada en timestamp + contenido
