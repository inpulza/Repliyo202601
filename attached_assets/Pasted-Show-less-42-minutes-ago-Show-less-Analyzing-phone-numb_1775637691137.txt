Diagnóstico del problema
3 causas raíz encontradas:

Mensajes del sistema de Facebook marcados como "inbound" — Cuando Facebook crea una conversación por un private reply, envía un mensaje como "Facebook created this chat because X commented on your post..." con una URL que contiene IDs numéricos largos (ej: id=100063745503440). Metricool trae estos mensajes como si fueran del cliente, y el sistema los procesa para extracción de teléfonos.

No se filtran URLs antes de extraer teléfonos — El servicio de enriquecimiento aplica regex directamente al texto completo del mensaje, incluyendo URLs que contienen secuencias de 10-15 dígitos (IDs de Facebook, IDs de comentarios, etc.).

El patrón regex \b\d{10,15}\b es demasiado amplio — Captura cualquier secuencia de 10-15 dígitos, sin importar si es un teléfono real o un ID numérico de Facebook.

Ahora voy a implementar la corrección. Los cambios serán:

Filtrar mensajes del sistema de Facebook antes del enriquecimiento
Eliminar URLs del texto antes de buscar teléfonos
Mejorar los patrones regex para ser más precisos

Resumen de lo encontrado
El problema exacto
Los mensajes del sistema de Facebook como:

"Facebook created this chat because Elianis Leyva commented on your post... See comment(https://facebook.com/story.php?...&**id=100063745503440**&comment_id=2310383216119922...)"

Eran marcados como mensajes del cliente (inbound), y la URL dentro contenía el ID de la página de Facebook de SSF (100063745503440). La regex de extracción capturaba los primeros 10 dígitos (1000637455) como un número de teléfono y lo normalizaba a +11000637455.

Evidencia: Encontré que tanto Elianis Leyva como Yudit Riego tenían exactamente el mismo teléfono falso +11000637455 — ambas tenían esos mensajes del sistema de Facebook.

Correcciones aplicadas
Detección de mensajes del sistema — Se agregó una función isSystemMessage() que identifica mensajes de Facebook en 3 idiomas (inglés, español, portugués) como "Facebook created this chat because...", "Auto-label added:", etc.

Eliminación de URLs antes de la extracción — Ahora se eliminan todas las URLs del texto antes de buscar teléfonos, para que los dígitos dentro de links nunca se capturen como números.

Se eliminó el patrón regex demasiado amplio — El patrón \b\d{10,15}\b que capturaba cualquier secuencia de 10-15 dígitos fue removido. Solo se mantienen los patrones que buscan formatos de teléfono reales.

Protección doble — Tanto el procesamiento en tiempo real (en syncService.ts) como el backfill están protegidos contra estos mensajes.

Limpieza de datos — Se limpiaron los 2 contactos que tenían el teléfono falso +11000637455.
