Proyecto: Conversation Lifecycle Management con Smart AI Closing para CRM Conversacional

Vas a continuar el desarrollo de un sistema completo de gestión del ciclo de vida de conversaciones para un CRM conversacional omnicanal. El sistema está completamente diseñado y documentado, y ahora entra en fase de implementación. Toda la documentación de referencia se encuentra en el archivo DOCUMENTACION_COMPLETA.md, específicamente en la sección titulada PRD-LIFECYCLE que comienza aproximadamente en la línea 5200. Esta sección contiene el Product Requirements Document completo con especificaciones de backend, frontend, base de datos y UX/UI. Además, existe una base de conocimientos en la carpeta docs/knowledge-base/ que contiene tres documentos markdown de referencia: CONVERSATION_LIFECYCLE_MANAGEMENT.md con el benchmarking de plataformas líderes como Intercom, Zendesk, Front y HubSpot; RESPOND_IO_TECHNICAL_ARCHITECTURE.md con patrones técnicos críticos como la Regla de Hierro de Reapertura y el flag ai_active; y CRM_DESIGN_SYSTEM_UX_UI.md con el sistema de diseño completo incluyendo el paradigma de layout de 3 paneles, anatomía de componentes y especificaciones de accesibilidad.

El sistema que vas a implementar consiste en una máquina de estados con cinco estados para las conversaciones: new (mensaje entrante sin respuesta), open (en proceso, asignado a agente o bot), pending (esperando respuesta del cliente), solved (propuesta de cierre con periodo de gracia configurable) y closed (archivado definitivo e inmutable). La transición de Solved a Closed ocurre automáticamente tras un periodo configurable por marca (default 24 horas), y la clave del diseño es que el estado Closed es irreversible: si un cliente responde a una conversación cerrada, se debe crear una nueva conversación referenciando la anterior, nunca reabrir la existente.

Existen patrones críticos de diseño que debes respetar estrictamente durante la implementación. Primero, la Regla de Hierro de Reapertura establece que un Agente de IA nunca puede reabrir una conversación cerrada, esto previene loops infinitos donde el bot responde "De nada" a un "Gracias" del cliente y crea un ciclo eterno de cierres y reaperturas. Segundo, el flag ai_active debe implementarse como un booleano en la tabla de conversaciones que se establece en false cuando se hace handoff a un agente humano, y el orquestador de IA debe verificar este flag antes de invocar cualquier LLM para evitar que bot y humano hablen simultáneamente. Tercero, el Thank You Detection es un servicio que analiza mensajes entrantes en conversaciones con estado Solved para determinar si son expresiones de gratitud (no reabrir) o nuevas consultas (sí reabrir), utilizando un árbol de decisión que verifica signos de interrogación, sentimiento de gratitud, y conteo de palabras menor a 15. Cuarto, las métricas de resolución deben separar siempre Bot Resolution de Agent Resolution porque mezclar promedios de 45 segundos del bot con 12 minutos del humano produce métricas inútiles.

La implementación debe seguir el siguiente orden de fases, que fue validado arquitectónicamente. La Fase 1 consiste en modificar el schema de base de datos en shared/schema.ts añadiendo los nuevos campos a la tabla conversations: status (text, default 'new'), closedAt, closedBy, closedByUserId, aiActive (boolean, default true), assignedToUserId, assignedAt, firstResponseAt, resolutionTimeMinutes, reopenCount, lastCustomerMessageAt, closingSummary, closingSentiment, closingIntent, y closingResolution. También debes crear dos nuevas tablas: conversationStatusHistory para el historial de cambios de estado con campos para conversationId, previousStatus, newStatus, changedBy, changedByUserId, reason y createdAt; y brandLifecycleSettings para configuración por marca con campos para solvedToClosedHours (default 24), thankYouDetectionEnabled, thankYouMaxWords (default 15), autoCloseInactivityHours (default 72), csatSurveyEnabled y csatSurveyDelayMinutes. Después de modificar el schema, ejecuta npx drizzle-kit push para aplicar los cambios a la base de datos y verifica que las migraciones se aplicaron correctamente consultando las tablas.

La Fase 2 consiste en crear el servicio de lifecycle en server/services/conversationLifecycleService.ts con métodos para markAsOpen, markAsPending, markAsSolved, markAsClosed, generateClosingSummary, closeWithSummary, analyzeIncomingMessage, shouldReopenConversation, calculateResolutionTime, y métodos para los cron jobs processAutoClose y processSolvedToClosedTransitions. Este servicio debe registrar cada cambio de estado en la tabla de historial incluyendo quién hizo el cambio (agent, bot, auto, customer) y el motivo. El servicio debe integrarse con el syncService existente para interceptar mensajes entrantes y evaluar si deben reabrir conversaciones en estado Solved.

La Fase 3 es el Thank You Detector en server/services/thankYouDetector.ts, un servicio que puede implementarse de dos formas: una versión básica con reglas heurísticas (verificar signos de interrogación, palabras clave de gratitud, longitud del mensaje) y una versión avanzada que use el LLM para análisis de sentimiento. El prompt para la versión con IA está documentado en la sección 3.6 del PRD-LIFECYCLE y debe retornar un objeto JSON con isThankYou, confidence, hasQuestion, hasNewRequest y reasoning.

La Fase 4 añade la generación de resúmenes con IA utilizando el prompt documentado en la sección 3.5 del PRD-LIFECYCLE, que debe generar un resumen de 2-3 oraciones, sentimiento (positive/neutral/negative), intent del cliente, resolución aplicada, topics y action items pendientes. El límite de contexto para resúmenes es de 100 mensajes (a diferencia de 20 para respuestas conversacionales), esto está documentado en la sección 2.8 del PRD.

La Fase 5 expone los endpoints de API documentados en la sección 3.4: POST /api/conversations/:id/status para cambiar estado, GET /api/conversations/:id/history para historial de cambios, POST /api/conversations/:id/close para cerrar con resumen IA, POST /api/conversations/:id/reopen para reapertura manual, POST /api/conversations/:id/generate-summary para generar resumen sin cerrar, PUT /api/conversations/:id/summary para editar resumen, y los endpoints de métricas GET /api/analytics/resolution-time, /api/analytics/reopen-rate, /api/analytics/csat y /api/analytics/attribution.

Las Fases 6-9 corresponden al frontend y están documentadas exhaustivamente en la sección 6 del PRD-LIFECYCLE con wireframes ASCII. La Fase 6 añade badges de estado en la lista de conversaciones en client/src/pages/Inbox.tsx. La Fase 7 implementa el modal de cierre con resumen editable (wireframe en sección 6.2). La Fase 8 es el dashboard de métricas (wireframe en sección 6.8). La Fase 9 integra encuestas CSAT inline.

Las medidas de seguridad y verificación que debes seguir durante la implementación son: antes de cada migración de base de datos, hacer backup conceptual revisando el estado actual del schema; después de cada cambio en el schema, verificar con una query SELECT que las columnas existen y tienen los defaults correctos; al implementar cada método del servicio de lifecycle, escribir primero el caso de uso y luego verificar manualmente con logs que las transiciones de estado funcionan; nunca exponer endpoints que permitan a un agente de IA ejecutar la reapertura de conversaciones cerradas, esto debe estar hard-coded como restricción en el servicio; verificar que el flag ai_active se actualiza atómicamente en la misma transacción que la asignación de agente; y antes de marcar cualquier fase como completada, revisar el código contra las especificaciones del PRD-LIFECYCLE para asegurar que no se omitió ningún requisito.

Para comenzar, te recomiendo leer primero la sección PRD-LIFECYCLE completa en DOCUMENTACION_COMPLETA.md (desde la línea 5200 aproximadamente hasta el final del documento), luego revisar el schema actual en shared/schema.ts para entender la estructura existente de la tabla conversations, y finalmente comenzar por la Fase 1 añadiendo los nuevos campos de forma incremental, verificando cada migración antes de continuar con la siguiente.
